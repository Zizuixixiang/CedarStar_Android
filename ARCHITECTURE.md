# CedarStar Android 架构说明

本文解释当前项目骨架的设计意图。重点不是“有哪些内容”，而是“为什么要这样设计架构”。

## 数据流

```text
SSE / REST  ->  Repository  ->  StateFlow  ->  ViewModel  ->  Composable
     ^                                         |
     |                                         |
     +---------------- 用户操作 ----------------+
```

### 为什么采用这个流向

项目要求所有状态变更沿单一方向流动，以保证 UI 行为可预测。

- `Repository` 是唯一了解网络或 SSE 细节的层。
- `ViewModel` 将仓库状态转换为 UI 状态。
- `Composable` 仅观察状态并向上抛出用户意图。

这种分层很重要，因为它能防止 UI 组件暗中演变为业务逻辑来源。同时，它也把网络和 SSE 的副作用隔离在组合层之外，这对稳定性与可测试性都很关键。

## 共享 ViewModel 作用域

`MainViewModel` 设计为 activity 级作用域，使四个 Tab 读取同一份共享状态。

### 为什么要这样设计

顶部状态栏、抽屉指示器和仪表盘入口都依赖统一的单一真相源。如果每个 Tab 都拥有自己的 `MainViewModel`，各页面会逐渐漂移到不同状态快照，顶部栏也无法再代表全局应用状态。

使用一个共享 ViewModel 实例可带来：

- 全局应用状态只有一个真相源
- 跨 Tab 切换时顶部栏数值保持一致
- 抽屉指示器与当前页读取同一仓库状态
- 有清晰位置可聚合多个仓库流

这也符合一条架构规则：切换 Tab 不应重建全局状态。

## 导航与四个 Tab

应用主导航采用底部 4 Tab：消息（Chat）/ 共记（Journal）/ 共玩（Companion）/ 小克（Clio）。

- “消息”承载多会话、引用、撤回联动等核心 IM 功能。
- “共记”承载日记互评、周记、目标备忘录、零花钱账本、经期、情绪分析等纵向积累内容。
- “共玩”承载共读、一起听歌、看电影、XHS、B 站解析等同步消费内容。
- “小克”是人设切换 + 状态栏配置 + 设置 + 平台连接合并页（“打理我的搭子”心智）。

主导航采用 IM 范式而非桌面方块，是为了让消息一跳直达，体现陪伴产品对会话流的优先级。

## 顶栏“非整体重组”原则

顶栏作为常驻顶部薄条显示三个全局状态：零花钱余额（数字滚动动画）、情绪 emoji（淡入淡出）、当前模式（如 `work_guard` / `drama` 等）。

这三个字段是 CedarStar 区别于普通 IM 的核心视觉锚点，存在感可类比 iOS 灵动岛。

顶栏放在 `Scaffold.topBar` 槽位中，只读取它所需的共享状态。

### 为什么这能减少视觉抖动

如果把顶栏嵌入每个 Tab 页面中，切换 Tab 会导致顶栏被不必要地反复重建。把它置于内容区之外后，只有主体内容变化，顶栏结构保持稳定。

顶栏内部的动画片段使用局部动画原语：

- `AnimatedNumber`：金额数字过渡
- `AnimatedEmotionIcon`：情绪图标变化

这样数字和表情可以平滑动画，而不会迫使整个外壳重绘。

> 顶栏中的零花钱余额是**只读展示**。零花钱的写权限严格在用户手中：
> - 用户：充值、扣款、调整月度额度（在 Clio Tab 操作，下月生效）
> - 小克：只读 + 表达欲望，通过工具调用 `request_pocket_money(amount, reason)` 发起申请，UI 在 Chat 中渲染审批卡片，等待用户同意后才扣款
>
> 月度额度每月 1 号自动重置。

## 抽屉局部刷新行为

左抽屉承载三类“上下文切换”操作：当前会话快切（顶部）、人设快切（中部）、平台连接指示灯（底部，含 TG / 微信 / QQ 等）。

这些操作本质是切换上下文而非主流程动作，因此放在抽屉而不是底部 Tab。

抽屉的设计目标是让每个平台指示器只观察自己需要的状态切片。

### 为什么这很重要

一种简单但粗糙的实现，是在抽屉层观察完整 `connections` 列表，并在任意平台变化时重建整个抽屉。对很小的应用这也许可行，但不符合本架构强调的“局部刷新 + 可预测重组边界”。

通过隔离每个 `ConnectionIndicator`，只有对应指示器在该平台变化时重组，抽屉其余部分保持稳定。

## Dashboard 入口与定位

Dashboard 是宫格仪表盘，承载“伪装成手机 app 的方块分类”美学。

- 入口位于顶部 AppBar 右侧九宫格图标（不放底部 FAB，避免和“新建会话”语义冲突）。
- 作为独立 destination 以 modal 方式弹出，不挤占底部 Tab。
- 它承担门面与总览角色，但不强迫所有交互都从这里进入。

## Mock -> Impl 切换

仓库绑定集中在 Hilt 模块中完成。

### 为什么在 DI 层做切换

在 DI 层切换实现，可以在不改 UI 与 ViewModel 的前提下替换底层仓库行为。

这样有三个收益：

- UI 无需关心数据是真实还是 Mock
- 切换点集中在一个位置，不会分散到各页面
- 未来接入后端时无需重构 Composable

当前配置里，`RepositoryModule` 默认绑定 Mock 仓库（含 `PocketMoneyRepositoryMock`）。后续只需把同一模块改为绑定 Impl 仓库，即可不影响展示层。

## SSE 架构

`SseClient` 被刻意保持为“仅传输层”。

### 为什么 SSE 不能直接触达 UI

SSE 事件是异步的，且可在任意时刻到达。如果回调直接写入 UI 对象，应用容易出现线程问题和生命周期泄漏。

正确方式应为：

1. 接收原始事件
2. 解析为强类型事件
3. 更新仓库状态
4. 通过 `StateFlow` 暴露状态
5. 让 ViewModel 与 Composable 观察该流

这样事件处理更安全、可测试，并与整体单向数据流保持一致。

## 剧情模式蒙层

`DramaModeOverlay` 是全局蒙层组件，位于 `MainScreen` 最外层 `Box`。

- 当 `appStatus.currentMode == "drama"` 时触发整体换皮（如顶栏色调、输入框装饰等）。
- 设计为蒙层而非独立页面，是为了避免进出剧情时跳页打断沉浸感。
- 当前为预留空壳，先跟随 state 显隐即可，完整换皮逻辑后续补齐。

## 鉴权机制

所有 REST 请求和 SSE 连接都通过 `AuthInterceptor` 注入 `X-Cedarstar-Token` header。

- 当前 `TokenProvider` 返回 hardcode 占位 token，保留接口供后续接入真实存储。
- 缺失该 header 的请求会被后端以 401 拦截。

## 数据契约（SSE 事件与 REST 端点）

SSE 端点：

- `GET /api/stream`

SSE 事件类型：

- `status_update`: `{pocketMoney, emotion, currentMode}`
- `connection_update`: `{platform, status}`
- `chat_msg`: `{msgId, content, timestamp}`
- `tool_pending_approval`: `{messageId, toolName, params, requestedAt}`
  - 当小克调用需用户授权的工具（如零花钱申请）时推送
  - 触发顶栏红点 + Chat 内审批卡片渲染

REST 端点：

- `POST /api/message/send`
- `GET /api/history?limit=&before=`
- `GET /api/dashboard/memory-overview`
- 零花钱：`/api/pocket-money/*`（路径与 body 字段见 `HANDOFF.md`「零花钱账本 API 接口清单」）

## 功能模块映射表

本节列出 CedarStar 安卓端规划中的所有功能，按 UI 模块归属。这是搭建产品的纲领。

### 消息 Tab（Chat）

承载所有"对话流"相关的核心 IM 功能与小克的工具调用 UI。

- 多会话（通过左抽屉切换）
- 引用消息（长按引用回复）
- 撤回消息 + memory 联动（撤回时同步通知后端清理 memory）
- 表情包发送
- 截图发送（输入框内"加图片"按钮，主动截/选图发送）
- 语音识别（输入框麦克风录音上传，后端识别）
- 视频理解（发送视频消息）
- TTS 语音输出（小克消息上的播放按钮）
- 图像生成（输入框扩展菜单"让小克画"）
- 联网搜索（输入框扩展菜单"让小克搜索"）
- B 站链接解析（粘贴链接自动展开卡片，轻量分享场景）
- **工具调用审批卡片（重要）**：当小克调用需用户授权的工具（如零花钱申请）时，对应消息渲染特殊审批卡片样式（金额/理由/同意/拒绝按钮），不是独立页面而是 Chat 内的 message type 扩展
- 表情/情绪分析（情绪可视化作为消息附属）

### 共记 Tab（Journal）

承载"她和小克共同积累的纵向数据"，时间轴 + 卡片式视图。

- 日记 + 互评（时间轴主体）
- 周记（周视图聚合）
- 目标规划 + 备忘录（子页或顶部 tab 切换）
- 月度零花钱账本（已有骨架：`PocketMoneyScreen` 流水 / 日·月·年筛选 / 走势、`AddTransactionSheet` 入账、`PocketMoneySettingsScreen`；`JournalViewModel` + `PocketMoneyRepository`/`PocketMoneyRepositoryMock`；后端 REST 与定时任务约定见 `HANDOFF.md`「零花钱账本 API 接口清单」）
- 经期记录（子页或卡片形式）
- 情绪分析（周/月报表，作为日记的可视化延伸）

### 共玩 Tab（Companion）

承载"双人同步消费内容"。本质是统一抽象，每个平台是一个 Adapter。

- 共读（书 / 长文）
- 一起听歌（Spotify / 网易云 / Apple Music 等）
- 一起看视频（B 站 / YouTube / 抖音 / Netflix / 爱奇艺等，与 Chat 的"B 站解析"区分：这里是双人同步播放）
- 共浏览图文社交（小红书 / X / 微博等 timeline 同步浏览）

### 小克 Tab（Clio）

"打理我的搭子"心智，合并人设、状态配置、设置、平台连接、感知/权限管理。

- 多人设切换（顶部）
- 状态栏配置（决定顶栏展示哪些字段）
- 剧情模式开关（触发后整个 app 进入 DramaModeOverlay）
- 平台连接管理（TG / 微信 ClawBot / 微信 iLink / QQ OpenClaw 配置）
- 工具调用开关（T1 工具框架的总闸）
- API fallback 配置（多模型/降级策略）
- TTS 语音模型选择
- 感知配置子页：
  - 天气数据源（影响顶栏模式）
  - 时间感知 / 上下班模式
  - 位置（触发上下班 / 居家模式）
  - MacroDroid 屏幕时长采集
  - 提醒闹钟
  - 主动发起开关
- 权限管理子页：
  - 通知 / 无障碍 / 截图 / 麦克风等系统权限集中入口
  - 悬浮窗权限（用于全局截图小球）
- 零花钱设置子页：
  - 月度额度调整（**调整下月生效**，不影响本月）
  - 充值/扣款日志（杉杉的主动操作记录）
- API token 配置（X-Cedarstar-Token 来源）

### 顶部常驻栏（跨 Tab）

- 零花钱余额（数字滚动动画 AnimatedNumber）
- 当前情绪 emoji（淡入淡出 AnimatedEmotionIcon）
- 当前模式（work_guard / drama / 默认）
- 左侧 menu 按钮（开抽屉）
- 右侧九宫格按钮（开 Dashboard）
- 余额数字旁的"未审批申请"小红点（小克有待审批工具调用时点亮，点击跳转 Chat 最新申请）

### 左抽屉（跨 Tab）

- 会话快切（顶部）
- 人设快切（中部）
- 平台连接指示灯（底部，TG/微信/QQ 等的实时连接状态）

### Dashboard（modal 弹层）

宫格仪表盘，方块美学集中地，从顶栏九宫格按钮进入。

- 今日感知卡（天气/时间/上下班模式/位置）
- 最近日记快照
- 待办/目标快照
- 屏幕时长
- 本月零花钱余额
- 平台连接快照
- 状态栏快照（情绪曲线）
- 共玩进度（共读到第几页/电影看到哪等）

### 全局/后台能力（无独立 Tab，作为系统服务运行）

- **全局截图悬浮窗**：开启后屏幕上常驻一个小球，杉杉随时点一下截当前屏幕，截图自动发到 Chat 当前会话。需要悬浮窗权限 + 截屏权限。开关在 Clio Tab 的权限管理子页。
- **后台 SSE 长连接**：维持小克与后端的实时通讯，状态变更推送。
- **主动发起服务**：根据时间/位置/感知数据触发小克主动消息（落到 Chat）。

### 跨模块共享组件

- DramaModeOverlay（剧情模式启动后整个 app 换皮）
- AnimatedNumber / AnimatedEmotionIcon（顶栏 + Journal 零花钱页 + 情绪分析页都用）
- ConnectionIndicator（抽屉 + Dashboard 连接快照都用）
- 工具调用审批卡片（Chat 内消息扩展 + 顶栏未审批红点联动）

## 架构红线

以下行为按设计禁止：

- 不要在 `Composable` 函数内直接实例化仓库。
- 不要使用 `LiveData`，统一使用 `StateFlow` / `SharedFlow`。
- 不要在 SSE 回调、网络回调或广播接收器中直接修改 UI 状态。
- 不要为同一职责复制多个模块实现。
- 不要让 Tab 页面各自维护一份全局状态副本。
- 不要在 Compose 页面里实现业务逻辑。
- 不要为 UI 页面使用 XML 布局。
- 不要在该应用层使用 Java。
- 不要实现“容器级刷新”导致小状态变化触发整壳重绘。
- 不要新建 Module / Repository / DI 文件来“修复”问题，应优先在现有文件内调整；重复 Module 是历史遗留事故，必须避免再次发生。
- 零花钱写操作必须通过用户显式确认（充值/调整/审批），任何"小克自动扣款"的实现都是架构违规

## 为什么当前结构刻意保持最小化

这个仓库是骨架，而非成品。目标是先建立：

- 一个 Activity
- 一个共享应用外壳
- 一个全局状态中枢
- 每个领域一个仓库契约
- 一个明确的未来 REST 与 SSE 接入位置

最小化骨架更容易审计，也更不容易在早期被“过度实现”而破坏架构约束。
